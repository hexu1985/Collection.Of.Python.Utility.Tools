Python打包：PyInstaller 教程

谭九鼎
一只普通的计算机爱好者

谈起Python打包，那必须谈PyInstaller，老牌经典。网上有无数文章介绍，然而在我看来大部分文章质量较差。

本文假定读者能在命令行中使用python和pip，懂虚拟环境的意思。

使用场景

Py打包的方式不止一种，PyInstaller具有以下缺点：

  • 它会把Python运行时和依赖的库打包进来。这对程序员来说没有任何意义，因为他们本来就装了环境，不需要再冗余一份
  • 打包的环境不能正常更新，任何改动都必须整体重新打包一遍
  • 打包后不能跨平台，且不能交叉编译
  • 体积大（相对库本身来说）

如果你这项目写出来是给其他程序员开发用，应选择setuptools。这也是Python社区最普遍、最正常的打包方式。它也能生成exe，只不过要从命令行调用罢了。详情可看我的上一篇打包文章：谭九鼎：Python打包：setuptools 与 setup.py/.cfg 入门简介

有两种情形适合使用PyInstaller。

一是给不会编程的人使用。普通用户期待的是双击运行程序，完。不能要求他们先去一个英文网页下载不知道有什么用的安装包，装完了还要在命令行里装依赖，再在命令行里使用。

二是部署到服务器上。如果只有这一个程序那就无所谓，但问题是这个系统中可能存在：

  • 依赖不同Python版本的项目
  • 依赖相同版本的Python，但不同版本的库
  • 别人开发的程序，你不知道它们依赖什么

对于第二种场景，理论上更合适的选择是Docker和k8s，但Win无法用，学这俩技术也有一定的负担，单机一般也用不上。也可以用虚拟环境加上管理Py版本的库，但终究只适合开发，不是用于Release的；因为虚拟环境不包含解释器真身，且不能移动位置。

第一种场景就是其他文章预定的场景了。有一说一确实，许多读者刚好就是这个需求。他们没有考虑过如何给其他程序员使用，源代码直接放着不打包；没听过setuptools，没想过用pip安装自己的库。甚至有的不会命令行，执行代码的方式是把.py关联到解释器，鼠标双击。

安装

一般在虚拟环境中安装PyInstaller，否则打包后往往体积较大：

python -m venv .venv
.venv\Scripts\Activate
(.venv) pip install pyinstaller

PyInstaller还会自动调用upx压缩减少体积，所以就把upx下下来放到Path里，过程略。

使用

项目需要有一个运行起点。我就不谈if __name__ == '__main__'了，喜欢把代码直接全写在全局的同学，多学学Python基础知识。

假设main.py是入口点，使用pyinstaller main.py，会进行打包并生成main.spec。之后再打包就是修改spec并pyinstaller -y main.spec了，不要再对main.py使用，否则会覆盖spec。

生成结果在dist文件夹中。构建过程在build文件夹中，warn-xxx.txt记录了找不到的模块。

如果Path里有upx，会自动使用。

默认是文件夹模式，虽然不是一整个单文件，但仍把所有.py编译成pyc打包进了一个主exe里。只要不改变依赖，部署时可以只更新主exe。如果软件要保存设置、生成日志，当然是文件夹模式方便。日常用的哪些程序不都是非单文件的么。

对于窗体，用-Fw变为单文件窗口模式。运行时解压到临时文件夹，不显示黑框框。确实对小白更友好，但请先确保程序作者——你，不是小白。另外根据文档的建议，单文件不适合需要提权的场景。

命令行参数

此处只列出编译期参数，像icon、datas等东西应在spec中指定，对spec使用许多参数也是忽略的。

  • -y：覆盖dist文件夹时不用交互式确认
  • --clean：默认有一定的增量生成功能，但如果出问题时可用此选项，尤其是开发早期依赖不稳定时
  • --noupx：不压缩以提高打包速度，用于测试时需要频繁生成
  • --log-level DEBUG：编译(打包)期显示详细信息
  • -d all：运行期显示详细的模块导入信息、不把py打包到一个文件中

spec

  • Analysis：根据自己的需要，主要修改datas和hiddenimports
  • EXE：Linux可开strip。icon也在这里
  • 模板是根据最开始的pyinstaller命令的参数生成的，对于-Fw可以考虑重新生成一下

datas

  • PyInstaller默认只打包.py，如果需要把Readme等复制过去，就使用datas；有的库带有数据文件也要用
  • 它是一个[(src:str, dest:str)]，把src复制到dest
  • src可有*通配符
  • dest可为.
  • 若src是个文件夹且想保留目录结构，需要重复写一遍：('folder', 'folder')，若写('folder', '.')就会解一层
  • 可用PyInstaller.utils.hooks.collect_data_files辅助
  • 没有exclude_datas的功能

hiddenimports

  • PyInstaller会试图分析imports语句，只打包用到了的，但有的动态import的不能自动分析出来，就要手动指定
  • 可用PyInstaller.utils.hooks.collect_submodules辅助

hooks

为了解决以上问题，PyInstaller实现了一个叫做hooks的功能，如果一个包存在对应的hooks，就会执行它们，获取datas和hiddenimports有哪些。关键是PyInstaller自带了一些库的hooks，这就是它所宣传的“compatible with 3rd-party packages out-of-the-box”。当然这些hooks仍是需要PyInstaller的开发者和社区贡献的，不同的库需要单独处理，可不是自动的。

相对路径

很多人打包后出现找不到图片图标这类问题，这本不应属于这篇文章负责的内容，但不了解的人实在太多。

假设c:\py\test.py中有一条open('data.txt')，运行它，会读取哪里的文件？是c:\py\data.txt吗？答案是无法确定。

相对路径相对的是哪里？它不是相对于正在运行的.py文件，也不是相对于项目根目录。它相对的是“工作目录”, Current Wording Directory, CWD。Shell中待输入的前面那个部分就是CWD，如C:\Users\Administrator>或者C:\py>。

只给文件路径，和CWD没有任何关系，自然无法确定相对的是哪里。只不过许多人在IDE中运行项目，那时的CWD是项目根目录。或者直接双击运行文件，看起来就是相对于打开的文件。但依赖这些行为都是错误的。

如果c:\py\test.py想读取c:\py\data.txt，应该用open(os.path.dirname(__file__) + '/data.txt')，其中__file__就表示的是当前正在运行的.py文件。或用pathlib的Path(__file__).with_name('data.txt')。

对于打包，又会引入新的变化因素，因为所有的py都打包到一个文件中了。

结论是，CWD仍然是不变的。__file__用于读取datas的内容，文件夹模式下效果看起来与不打包相同。但单文件模式下，数据会释放到临时文件夹中，它也变到了临时文件夹中，仍能读取datas，却不能表示那个单exe所在的目录。此时要用sys.executable或sys.argv[0]。

这三者无法相互替代，想清楚到底要从哪里读取数据。如果不理解这些，乱用相对路径，打包出来当然各种问题；但这是写程序的人的问题，不是PyInstaller的问题。甚至Python还缺少一种从项目根目录下读取数据的方式。

体积

都用Python了，都打包运行时和库了，那就不必过分纠结体积，只要保证不是异常大就可以了（常见于Anaconda）。PyInstaller能分析依赖，不是无脑复制。虚拟环境只装需要的库，再加上upx压缩，还可以再在外面用7z压缩一部分。

Hello World的大小是7.5MB左右，7z后是5.32MB。这个大小已经比embeddable版本小了，更不论可维护性和库兼容性了。

我还看到有人说“建议用GCC”、“巧用visual studio（VC++）解决pyinstaller打包的exe文件过大的问题”。我想这难道说的是Cython？仔细一看，是用C的system()调用命令行，路径还是写死的，然后评论区有人说“非常牛X，直接从40M到40k”，看得我人都傻了。
